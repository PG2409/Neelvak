A complete diagnostic audit of the Neelvak AIOS execution pipeline was

Primary symptom: every prompt immediately returned cached garbage or

Root cause chain:

3. MEMORY ANALYSIS
==================
File:   memory/manager.py  Method: _calculate_css

ORIGINAL (broken):
    jaccard = |intersection| / |union|
    overlap  = |intersection| / |prompt_tokens|   <- asymmetric
    css      = 0.5 * jaccard + 0.5 * overlap

The overlap term equals 1.0 whenever ALL prompt tokens appear in a cached key,
regardless of how long the key is. "Hello" (1 token) matched any 500-word key
containing "hello" -> overlap=1.0 -> css=0.50+. With any nonzero Jaccard this
exceeded the STANDARD threshold (0.85) and triggered a false cache hit.

FIXED:
    return |intersection| / |union|   (pure symmetric Jaccard)

Jaccard is bounded [0,1] and symmetric. "Hello" vs 500-token key -> 0.002.
Never exceeds any threshold unless prompts are genuinely similar.

Secondary: l2_manifest.json cleared (400+ stale entries removed).

13. REMAINING RISKS
===================
1. MicroRuntime (C) does not call LLM for batch slices -- returns
   "Resolved {item}". By design for batch workflows; acceptable.

2. CompetitiveRuntime Watching Agent Pass 2 uses output length as quality
   proxy rather than a true semantic LLM judge. For equal-length responses,
   Worker X is selected by default.

3. Provider timeouts (25s) may trigger for large prompts on slower models.
   Node-level timeout is 30s. Overlap window is 5s.

4. The CSS threshold (0.85 Jaccard) means two prompts with ~85%+ token
   overlap will cache-hit on subsequent requests. This is the desired
   behavior but should be documented for users.
